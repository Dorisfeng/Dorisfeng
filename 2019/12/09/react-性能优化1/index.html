一个组件重新重新渲染，一般三种情况：

  • 组件自己的状态改变
  • 父组件重新渲染，props 没有改变，子组件重新渲染

  • 父组件重新渲染，props 改变，子组件重新渲染

    结合当前情况，子组件没变，第一条不符合，用了React.memo，props 不改变，子组件不会重新渲染，第二条也不对，所以只能是第三条。
    props 传递name,count,callback，前两个是常量不会改变，那只能是 callback 变了，函数式组件每次重新渲染，都会从头开始执行，那么两次创建的 callback 内存地址肯定发生变化，所以子组件重新渲染。
    解决方法：
    在函数没有改变的时候，重新渲染的时候保持两个函数的引用一致。

2. 减少计算量

主要是减少重复计算，对于函数组件来说，每次 render 都会重新开始在执行函数调用

每次点击+1 都会重新渲染组件，执行 expensiveFn 函数，并输出49995000，想办法把计算出来的值缓存起来，每次调用函数返回缓存值就可以了。

  • useMemo

useMemo 的第一个参数就是一个函数，这个函数返回的值会被缓存起来，同时这个值会作为 useMemo 的返回值，第二个参数是一个数组依赖，如果数组里面的值有变化，那么就会重新去执行第一个参数里面的函数，并将函数返回的值缓存起来并作为 useMemo 的返回值。
优化后代码

useMemo 主要用来缓存计算量比较大的函数结果，避免不必要的重复计算。
注意

  • 如果没有提供第二个数组参数，useMemo 每次渲染都会计算新的值。
  • 计算量如果很小的计算函数，不建议使用useMemo，因为这点优化并不会作为性能瓶颈的要点，反而可能使用错误还会引起一些性能问题。

不要使用内联函数定义

使用内联函数，每次render函数执行都会创建一个新的函数实例，增加了 React 应用的工作量。

避免使用内联样式属性

使用内联样式属性需要花费更多时间来处理脚本和渲染，因为它必须映射传递给实际 CSS 属性的所有样式规则

列表中使用唯一键

键可以用来标识已更改，添加或删除项目，键为元素提供稳定的标识，一个键对应列表中的唯一一个元素。
index 作为键不会出现标识不一致的情况。
以下情况可以使用 index 作为键:

  • 列表是静态的，项目不随时间变化
  • 数据结构中没有提供 ID
  • list 永不重新排序或过滤
  • 不会从顶部或中间添加或删除项目

使用 index 作为键，如果在列表中添加项目，会加大错误率并降低应用的性能。因为在表头添加新项目后面所有的列表项的键值都会+1，导致性能下降。
总结：

  • Key 不仅影响性能，更重要的作用是标识，随机分配和更改的值不能算作标识
  • 根据建模方式提供合适的键值，如果没有 ID，建议使用 hash 函数生成 ID
  • 数组已经有了内部键，但是插入新元素时，这些键就会变化
    可以使用全局定义一个var localCounter = 1;来添加稳定且唯一的 key 值。

事件节流和防抖

用来限制指定时间内调用事件处理程序的数量。

  • 节流 throttle
    指连续触发事件但是在 n 秒中只执行一次函数。节流会稀释函数的执行频率。
    比如，页面滚动，不会频繁的触发滚动事件，而是将事件延迟一段时间以便将多个事件堆叠在一起执行。确保函数在特定时间段内至少调用一次。 在处理无限滚动并且当用户接近页面底部必须获取数据的时候可以使用节流，否则滚动到底部触发多个事件，触发多次网络调用，从而导致性能问题。

  • 防抖 debounce
    触发事件后，延迟 n 秒执行函数，如果此时间内继续触发事件，重新计算延迟时间。
    比如用户停止输入 3 秒钟之后触发查询事件

    • 立即执行版的：触发事件后函数会立即执行，然后 n 秒内不触发事件才能继续执行函数的效果。
    • 非立即执行：触发事件后，n 秒后执行函数

使用 CDN

这些 CDN 是可以在你的应用中使用的外部资源，也可以创建私有 CDN 并托管我们的文件和资源。

  • 不同的域名，浏览器限制了单个域名的并发连接数量，资源在不同域名下面，提高并发能力。
  • 文件可以已被缓存。很多网站使用这些 CDN，可能你需要的资源已经被缓存在浏览器中了
  • 高容量基础设施。大公司托管，遍布全球，可以向最近的数据中心请求服务，从而减小延迟

使用 css 动画代替 JS 动画

  • 如果 CSS 可以实现某些 JS 功能，那就用 CSS。
  • 如果 HTML 可以实现某些 JS 功能，那就用 HTML。
    原因如下：
  • 破损的 CSS 规则和样式不会导致网页损坏，但是 JS 会
  • 解析 CSS 非常便宜，它是声明性的，我们可以为样式并行创建内存中的表达，可以推迟样式属性的计算，直到元素绘制完成。
  • 动画加载 JavaScript 库的成本相对较高，消耗更多网络带宽和计算时间。
  • 虽然 JavaScript 可以提供比 CSS 更多的优化，但优化过的 JavaScript 代码也可能卡住 UI 并导致 Web 浏览器崩溃。

在 web 服务器上启用 gzip 压缩

压缩是节省网络带宽和加速应用的最简单方法。
Gzip 是一种能够快速压缩和解压文件的数据压缩算法。
可以压缩几乎所有类型的文件，例如图像、文本、JavaScript 文件、样式文件等。
当 Web 服务器收到请求，会提取文件数据并查找 Accept-Encoding 标头以确定如何压缩应用。
如果服务器支持 gzip 压缩，资源会被压缩后通过网络发送。每份资源的压缩副本（添加了 Content-Encoding 标头）指定使用 gzip 解压。
压缩和解压缩文件属于 CPU 密集型任务，但我们还是建议对网页使用 gzip 压缩。

Worker 在后台运行，可以在不中断主线程的情况下执行多个脚本和 JS 任务。

React 组件的服务端渲染（还待研究）

服务端渲染可以减少初始页面加载延迟。
服务端渲染是指第一个组件显示的内容是从服务器本身发送的，而不是浏览器级别操作。
优点：

  • 性能：初始页面内容和数据是从服务器本身加载，不需要添加加载器和下拉列表，而是等待初始页面加载完毕后再加载初始组件。
  • SEO（搜索引擎优化）：爬虫在应用初始加载时查找页面内容。在客户端渲染时初始 Web 页面不包含所需的组件，这些组件需要等 React 脚本等文件加载完毕后才渲染出来。
